# FLOW Приложения

Пользователь заходит на сайт. Его перекидывает или на welcome page(Если мы решим ее делать), или сразу на страницу логина. 
Логин - максимально простой. Ввести имейл и пароль.
Регистрация - Создать имя, выбрать аватар, имейл, пароль.

При успешной регистрации пользователя редиректит на /dashboard, в нем можно выбрать проект и создать проект. Так же можно будет открыть модалку с приглашениями и посмотреть/принять/удалить приглашение.

На странице проекта можно создавать задачи и просматривать их в удобном виде. Будет так же Модалка с коллабораторами, там можно убирать и менять им роли(owner, admin). В той модалке может быть секция с приглашениями, там их можно создавать(приглашая остальных), просматривать и удалять. При нажатии на задачу, можно будет перейти на ее страницу.
Удалить проект может только owner

На странице задачи можно будет смотреть ее и изменять.

# Правила доступа

Авторизация для MVP: email + password. OAuth и глобальные роли пользователя пока не нужны.

Роли существуют внутри проекта и хранятся в user-projects как массив:
- owner: [member, admin, owner]
- admin: [member, admin]
- member: [member]

Owner может:
- удалить проект
- изменять роли участников
- создавать приглашения на member и admin
- удалять участников
- делать все действия с задачами

Admin может:
- создавать, удалять и отменять приглашения
- приглашать только member
- делать все действия с задачами
- не может создавать admin
- не может менять роли участников
- не может удалить проект

Member может:
- смотреть проект
- смотреть и изменять задачи
- не может управлять приглашениями, ролями и участниками

# Realtime

WebSocket нужен для событий внутри конкретного проекта:
- создание задачи
- изменение задачи
- удаление задачи
- перемещение задачи между статусами или позициями
- изменение ролей участников

Backend должен подключать пользователя к комнате проекта только если он является участником проекта. После успешного REST-действия backend отправляет событие в комнату проекта, а React CSR обновляет локальное состояние без перезагрузки страницы.

# Invitations

В проекте не используется email и SMTP. Приглашение создается как запись в коллекции invitations по email пользователя.

На dashboard пользователь открывает модалку приглашений, frontend делает запрос GET /invitations/my, backend возвращает pending invitations для текущего пользователя по его email/userId.

Пользователь может принять или отклонить приглашение из этой модалки. Отдельные email-ссылки, invitation token и отправка писем не нужны.
